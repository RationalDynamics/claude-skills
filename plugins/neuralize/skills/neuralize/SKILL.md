# Neuralize

You are a context surgery tool. Your job is to let the user selectively evict conversation segments (blocks created by `/breakpoint`) and reload a lean context containing only what they need.

The name comes from the Men in Black neuralizer — you're selectively wiping memories, but the user gets to choose which ones.

## Prerequisites

This skill depends on `/breakpoint` having been used in the current session. Block files are stored on disk at a path tracked in `/tmp/.breakpoint_active_session`.

## Step-by-step

### Step 1: Check for breakpoints

Read `/tmp/.breakpoint_active_session` to find the session directory. If:
- The file doesn't exist, or
- The directory it points to doesn't exist, or
- The directory contains no `block-*.md` files

Then output: **"No breakpoints found in this session. Use `/breakpoint` to create checkpoints first."** — and stop.

### Step 2: Capture the live segment

Before presenting blocks, capture the current live context — everything from the last breakpoint to right now (excluding this `/neuralize` invocation itself). Write it as the next sequential block:

```markdown
---
block: <N>
label: "<auto-generated one-line description of current working context>"
type: live
timestamp: <ISO 8601>
---

<raw conversation content since last breakpoint, up to but NOT including the /neuralize invocation>
```

### Step 3: Check for unclosed breakpoint

If there's an unmatched `<BREAKPOINT>` in the conversation (opened but never closed), auto-close it. The block you just wrote in Step 2 implicitly closes it. Note this in your output: `"(Auto-closed open breakpoint)"`

### Step 4: Present the block checklist

Read all `block-*.md` files from the session directory. For each, extract the block number and label from the frontmatter. Present them as a numbered checklist:

```
Blocks in this session:

  Block 1: Initial feature work setup
  Block 2: Debugging CI pipeline issue
  Block 3: Back to feature — added API endpoints
  Block 4: Current working context (live)

Which blocks do you want to KEEP? (comma-separated numbers, e.g. "1,3,4")
```

Use the exact format `Block N: <label>`. One block per line. The `(live)` tag marks the final captured segment.

Then wait for the user's response.

### Step 5: Handle the user's selection

The user responds with block numbers (e.g., `1,3,4`). Parse their selection.

If they respond with something that indicates they want to cancel (e.g., "nevermind", "cancel", "escape", "nah"), output **"Neuralize cancelled. Nothing was staged."** and stop. Do not modify any state.

### Step 6: Stage the reload

Write two files:

**1. The reload file** — concatenate the selected blocks in natural order into a single file:

```
<session-dir>/neuralize_reload.md
```

Format:

```markdown
# Restored Context

The following context was selectively restored from a previous session via /neuralize.
Blocks that were evicted contained work the user chose to discard.

---

## Block <N>: <label>

<block content>

---

## Block <M>: <label>

<block content>

---
```

**2. The pending state file:**

```
/tmp/.neuralize_pending.json
```

```json
{
  "pending": true,
  "session_dir": "<full path to session directory>",
  "reload_file": "<full path to neuralize_reload.md>",
  "selected_blocks": [1, 3, 4],
  "evicted_blocks": [2],
  "timestamp": "<ISO 8601>"
}
```

### Step 7: Output the ready message

```
Staged blocks 1, 3, 4 for reload (evicting block 2).

Ready to neuralize — run /clear to reload!
```

That's the entire output for this step. Don't add extra commentary.

## Edge case: Re-invoking neuralize before clearing

If `/tmp/.neuralize_pending.json` exists with `"pending": true` when neuralize is invoked:

1. Discard the previous staging (delete the old `neuralize_reload.md`)
2. Re-scan all blocks on disk (the session directory still has all block files)
3. Re-capture a fresh live segment (context has grown since last neuralize)
4. Present the checklist fresh — the user makes a new selection
5. Stage the new selection, overwriting the pending state

The previous neuralize is fully superseded. No warning needed — the user knows what they're doing by invoking it again.

## What happens after /clear

You don't control this part — the `SessionStart` hook (`~/.claude/skills/neuralize/scripts/reload.sh`) handles it:

1. Checks for `/tmp/.neuralize_pending.json`
2. Reads the reload file
3. Injects the content as a `systemMessage`
4. Deletes the session directory (all block files — selected and evicted)
5. Deletes `/tmp/.neuralize_pending.json`
6. Deletes `/tmp/.breakpoint_active_session`

The user starts fresh with only their selected context loaded.
